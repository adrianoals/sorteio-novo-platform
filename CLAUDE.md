## What This Project Is

SorteioNovo V2 — a multi-tenant SaaS web app for managing condominiums and running parking spot lottery draws. Built with Next.js App Router, PostgreSQL (Drizzle ORM), and NextAuth.

Production: https://app.sorteionovo.com.br

## Commands

```bash
npm run dev          # Development server (http://localhost:3000)
npm run build        # Production build
npm run lint         # ESLint (Next.js + TypeScript)
npm run db:generate  # Generate migration from schema changes
npm run db:migrate   # Apply pending migrations
npm run db:studio    # Drizzle Studio (DB browser)
npm run db:seed      # Seed dev data (1 tenant + test data)
docker-compose up -d # Start PostgreSQL (dev only, port 5432)
```

Type-check: `npx tsc --noEmit`

## Environment Variables

Required: `DATABASE_URL`, `NEXTAUTH_SECRET`, `NEXTAUTH_URL`
Optional: `DB_POOL_MAX` (pool size, default 5 prod / 10 dev)

Copy `.env.example` → `.env.local` for local dev.

## Architecture

### Multi-Tenant Model
Every data table has `tenant_id`. All queries MUST filter by `tenant_id` for data isolation. Tenants are identified by UUID internally and by `slug` in public URLs.

### Route Layout Groups
- `src/app/admin/(with-header)/` — Admin pages with header/nav (tenant list, tenant detail with tabs)
- `src/app/admin/(clean)/` — Minimal layout pages (draw execution, draw results view)
- `src/app/[slug]/` — Public tenant pages (draw results accessed via QR code)

### Data Flow Pattern
- **Server Components** (`page.tsx`) prefetch data from DB via Drizzle, pass as SWR fallback
- **Client Components** (`*-tab.tsx`, `*-client.tsx`) use `useSWR` with the prefetched fallback for instant rendering
- After mutations (create/update/delete), call `mutate()` to revalidate SWR cache
- SWR fetcher is at `src/lib/swr.ts`

### Tenant Detail Page (main admin page)
`page.tsx` loads tenant + all tab data in a single `Promise.all`, passes as `fallback` to `<SWRConfig>` in `tenant-tabs.tsx`. Each tab uses `useSWR(apiUrl, fetcher)` which picks up fallback data instantly.

Tabs: Config, Blocos, Apartamentos, Vagas, Importações, Sorteios, Status.

### API Routes
REST-style under `src/app/api/admin/tenants/[id]/`. Each resource has `route.ts` with GET/POST and `[resourceId]/route.ts` with PATCH/DELETE. Context endpoints (`apartments/context`, `spots/context`) return combined data for tabs.

### Database
- ORM: Drizzle (`src/db/index.ts` for client, `src/db/schema/` for schema)
- Schema: one file per entity (tenants, blocks, apartments, parking_spots, draws, draw_results, audit_logs, users)
- Relations defined in `src/db/schema/index.ts`
- Migrations: `src/db/migrations/` (auto-generated by `db:generate`)
- Config: `drizzle.config.ts`
- Connection: postgres-js driver with `prepare: false` (required for Supabase pooler)

### Auth
NextAuth 5 beta with Credentials provider. JWT sessions (30-day). Config in `src/auth.ts`. Login page: `/admin/login`.

### Validation
Zod schemas in `src/lib/validations/`. Used in API routes via `schema.safeParse(body)`.

## Key Conventions

- Import alias: `@/*` → `./src/*`
- Formatting: 2 spaces, double quotes, TypeScript strict mode
- Styling: Tailwind CSS 4, utility-first, no CSS modules
- Client components: `"use client"` directive, suffix `-tab.tsx` or `-page-client.tsx`
- All DB read endpoints should include `Cache-Control: private, s-maxage=30, stale-while-revalidate=60`
- Parallelize independent DB queries with `Promise.all`
